ChainPilot · M6-T4

S1 诊断 Agent 调用流程

回答"这个物料为什么缺？"——LLM 在一个有界循环里自主调用只读工具、顺藤摸瓜查病因， 终点结论必须通过证据护栏。任何一步失败都有确定性降级路径：宁可说"未定位"，绝不编造。

max 8 步 temperature = 0 ReAct 文本协议 DuckDB 只读 零写库

01主流程 · 有界循环

入口：待诊断物料 material_pn
来自风险页的红色物料（如 GT-01 PN-00211）· CLI python -m agent.diagnose，T4b 将接入物料详情页按钮
组装 System Prompt
5 个工具的说明 + 5 个归因类别枚举 + JSON 输出契约——全部从代码常量生成，单一来源，prompt 与代码不可能漂移
有界循环 max 8 步 · 每步一次 LLM 调用 · 第 8 步强制收敛
observation 回填对话 ↻
①LLM 输出下一步动作（DeepSeek）
只允许一个 JSON 对象：{"thought", "action", "args"}，或 {"action": "final", "category", "root_cause"}
②解析器提取 JSON
代码解析，不靠模型自觉遵守格式
解析失败 → 纠错重试 ×1 把错误信息喂回模型再给一次机会；连续两次失败 → 终止为 unknown 降级结论
③分发：action 是工具还是 final？
工具 → 在只读连接上执行（见 02 节工具箱）；final → 携带结论跳出循环
action = final → 跳出循环 进入终点护栏（03 节）
④observation 追加到对话，回到 ①
工具返回的紧凑文本渲染成为模型下一步推理的证据
第 8 步仍未 final → 强制收敛 category = unknown，root_cause 由代码从 trace 确定性拼出"未定位，已排除 X / Y"
final: category + root_cause
终点证据护栏 verify_answer
全部工具 observation 拼成合成证据集；结论里的每个数字必须能在其中找到——与周报护栏同一套代码、同一个哲学
PASS
DiagnosisResult
category · root_cause（人话归因，引用真实数字）· 完整 trace · token 用量
FAIL → 降级
确定性模板替换
root_cause 换成按 category 的模板句，数字全部取自 trace；degraded = True 如实标记

02工具箱 · 5 个只读工具

get_risk_detail
当期风险行 + 供应画像（供应商数 / 最短交期 / 集中度）+ 最新库存
get_po_status
在途 PO 明细，按"缺口日前 / 地平线内 / 地平线外"分桶——追料够不够一眼可判
get_shared_demand
地平线内消耗该料的 SKU 需求分解——共用料竞争的直接证据
get_forecast_error
模型级预测精度摘要（粒度粗，仅够排除"预测不可信"）
query_sql
兜底自由查询，包 execute_safe——白名单表、强制 LIMIT、只读

全部工具 → DuckDB read_only 连接，Agent 在架构上不具备写库能力

03对照组 · workflow 决策树（人类计划员 SOP 写死成代码）
